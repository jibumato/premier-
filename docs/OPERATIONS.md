# プルミエ！ 運用マニュアル（ワンオペ運営）

> 「誰が・いつ・どう対応するか」を決めた運用ルール。**仕組み（自動処理・自動バッジ・通報の
> 自動非表示）は実装済み**なので、本書は**人手で回す部分の判断基準と頻度**を定めます。
> 技術的な手順（SQL・R2削除コマンド等）は [SETUP.md](SETUP.md) を参照。

- **更新日**: 2026-07-11 ・ 版: ops v1.0
- **運営者**: Type&Co（屋号） ・ お問い合わせ: 16typeandco@gmail.com
- **前提**: ひとり運営（ワンオペ）。手作業を最小化し、自動処理で回す設計。

## 0. 運用の全体像

| いつ | やること | 所要 |
| --- | --- | --- |
| **毎日（1回・5〜10分）** | 通報キュー・本人確認申請・問い合わせメールの確認 | 短い |
| **週次（1回・15〜30分）** | 重大通報の棚卸し、自動非表示になった投稿の妥当性チェック、指標確認 | 中 |
| **随時（通知が来たら）** | 未成年・犯罪等の重大案件は最優先で即対応 | 都度 |

> 基本方針：**軽微なものは自動処理に任せ、人が見るのは「重大・例外・グレー」だけ**。
> 迷ったら「ユーザーの安全 > 表現の自由 > 運営の手間」の順で判断する。

## 1. 本人確認（KYC）の手動レビュー運用

ユーザーが「設定 →本人確認 →身分証アップロード →申請」すると `verification_requests`
（status=`pending`）が作られます。**手順の詳細（承認/却下のSQL・画像削除）は
[SETUP.md 本人確認の運営手順](SETUP.md) に記載。** ここでは運用ルールを定めます。

### 確認頻度
- **1日1回**、`pending` を確認する（申請は多くないので基本これで十分）。
- 申請から**48時間以内**の承認/却下を目安（SLA）。遅れそうなら気にしすぎない ── β/初期は件数が少ない。

### 判断基準（承認の目安）
- ✅ **承認**：身分証の顔写真が鮮明／氏名・生年月日が読める／**18歳以上**が確認できる。
- ❌ **却下**：画像が不鮮明・一部隠れて年齢が確認できない／別人・加工が疑われる／マイナンバーの番号が写っている（※番号は取得しないので、番号を隠して再提出を案内）。
- ⚠️ **18歳未満と判明**：却下し、規約3章に基づきアカウントの利用停止を検討（本サービスは18歳以上限定）。

### 個人情報の取扱い（厳守）
- **確認が終わったら身分証画像を必ず削除**（R2 の `kyc/<user_id>/...`）。確認済みフラグだけ残す。
- マイナンバーカードは**番号を取得・保管しない**（顔写真面のみ）。
- この運用は[プライバシーポリシー §4](PRIVACY_DRAFT.md) と整合させること。

## 2. 通報対応の運用

通報は**自動処理が基本**です。一定数の通報が集まると DB トリガー
（`process_report_threshold`, SECURITY DEFINER）が対象を**自動で非表示**にします
（`content_flags`）。ブロックした相手の投稿も各画面で自動的に見えなくなります。
**運営が人手で見るのは以下だけ：**

### 随時（最優先・即対応）
次のカテゴリの通報は、閾値を待たず**発見/通知しだい最優先で対応**：
- **未成年に関する性的コンテンツ**（実在・創作を問わず）→ 即削除。必要に応じ関係機関へ通報。
- **犯罪・脅迫・自傷他害の予告** → 即対応。深刻なら警察等へ。
- **なりすまし・晒し（個人情報の無断公開）** → 確認のうえ削除。

### 週次（棚卸し）
- 自動非表示になった投稿を一覧で見て、**行き過ぎた非表示（濫用通報）がないか**を確認。
  誤りがあれば復帰させる（`content_flags` の該当行を解除）。
- 同一ユーザーへの通報が繰り返されていないか（規約違反の常習）を確認し、必要ならアカウント制限。

### 判断のグレーゾーン
- **露出の目安**：下着／下着と誤認される衣装がそのまま写る、性的部位を強調する構図は不可
  （衣装表現の範囲での露出はインナー等で処理されていることが前提）。→ 規約 第2章A に準拠。
- 迷ったら**まず非公開化してから本人に確認**（削除は最終手段）。

### メッセージ（DM）の確認（通報対応のみ・監査ログ必須）
嫌がらせ・詐欺・脅迫などの通報で、DM 本文を確認しないと判断できない場合に限り、運営は
該当ユーザーの会話を確認できます（設定 → 運営 → アカウント対応 → 対象ユーザーの「詳細」→
「メッセージの確認」）。**通信の秘密・プライバシーに関わるため、次を厳守：**

- **通報のあるユーザーの会話しか開けません**（DB 側で強制。通報が無ければ列挙も閲覧も不可）。
  無差別な閲覧・巡回はしないこと。
- 会話を開くと**必ず理由の入力**を求められ、**誰が・いつ・どの会話を・なぜ見たか**が
  監査ログ（`admin_message_access_log`）に残ります。ログの改ざん・削除はできません
  （設定 → 運営 → 「メッセージ閲覧の記録」で全運営が相互確認可能）。
- 目的外（好奇心・私的関心）での閲覧は禁止。必要最小限の範囲にとどめること。
- 実装は `messages` の RLS ではなく SECURITY DEFINER 関数経由（`is_admin()` 限定・0057）。

> ⚠️ **前提**: この機能を実際に使う前に、**プライバシーポリシー/規約で「通報対応・安全確保の
> ため、必要な範囲で運営がメッセージを確認する場合がある」旨を開示し同意を得る**こと。
> 実装（本機能）だけでは通信の秘密の観点で法的根拠を満たしません。開示文の追記は別タスク。

## 4. エスカレーション基準（人へ相談する線引き）

ワンオペでも、次は**自分だけで判断せず外部に相談**する：
- 法的リスクのある通報（名誉毀損・著作権侵害の削除要請、開示請求、警察・弁護士からの連絡）
  → **弁護士へ相談**。
- 大量の不正登録・スパム・攻撃 → 一時的に新規登録を絞る等の技術対応を検討。
- 決済（投げ銭）導入後の金銭トラブル → 決済事業者（Stripe）の規約・サポートに沿って対応。

## 5. 運用を支える実装（参考）

| 仕組み | 実装 | 人手の関与 |
| --- | --- | --- |

## 6. データを失わないために（バックアップ方針）

老舗コスプレSNSの障害騒動（「置きっぱなしの写真が消えるかも」）の教訓として、
**ユーザーの写真と応募履歴は預かりもの**という前提で運用します。

### 6-1. データベース（Supabase）

- Supabase は有償プランで**日次自動バックアップ**（Pro で 7日保持、PITR はアドオン）。
  無料プランのあいだは自動バックアップが無いため、**月1回以上**ダッシュボードの
  Database → Backups もしくは `pg_dump` で手動エクスポートし、ローカル/別クラウドに保管する。
- スキーマはリポジトリの `supabase/migrations/` が正本（コードでいつでも再構築可能）。
  **失って困るのはデータ**なので、エクスポート対象はデータ本体。

### 6-2. 画像（Cloudflare R2）

- R2 は高耐久ストレージだが、**誤削除・バグによる削除は防げない**。
  余裕ができたら R2 の「バケットのバージョニング」または定期的な
  `rclone sync`（R2 → 別バケット/別サービス）を設定する。
- アプリ側は「削除＝R2ファイルも削除」の方針（A方式）なので、
  ユーザー操作による削除は仕様どおり。守るのは**運営側の事故**から。

### 6-3. 障害時のユーザー告知

- 障害・メンテナンスの際は「お知らせ」（`announcements`）で状況と復旧見込みを掲示する
  （手順は [ANNOUNCEMENTS.md](ANNOUNCEMENTS.md)）。沈黙が一番信頼を損なう。
